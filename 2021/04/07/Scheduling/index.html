Scheduling

LLVM IR -> selectionDAG生成DAG

这个DAG的信息还十分缺少，我们会对他进行pre-scheduling

然后变成MachineInstr 在这个过程中会进行 Scheduling

之后会进行 RA，RA之后会进行post-Scheduling

这里记载一下 machineInstr的Scheduling；

关于scheduling在计算机组成原理里应该有了解过：

指令流水线

————》

​ ————》

​ ————》

指令流水会出现三种问题：

数据依赖（读写依赖，写读依赖，写写依赖）

？？依赖（不记得名字了，可以举例：比如加法器不够用，你却连着来两条mul，这谁顶得住。）

branch依赖？（跳转影响？）

有了流水线的概念，就是我们要怎么安排让指令流水线起作用，达到加速；

拿出例子

给出指令格式： opcode dst op0 op1 op2

给出指令cycle数：

load 4 cycle

add 2 cycle

mul 3 cycle

store 4 cycle

来一段简单指令：

load R1 C0 4

load R2 C1 4

add R3 R1 R2 2

load R4 C2 4

mul R5 R3 R4 3

load R6 C3 4

mul R7 R5 R6 3

load R8 C4 4

mul R9 R7 R8 3

store R10 R9 4

（好家伙，好像是让我把latency拉满了吗？）

演示一下

latency-first and bottom-up

将IR 转成DAG

从底部开始，latency-first

第一条：

store

第二条：

mul

第三条：注意这个地方有两个predecessor： mul 和 load

我们是latency-first mul 3 < load 4

所以我们选择

mul

第四条：

mul用了3 ，load的latency更新为 4-3 = 1

这次比较 mul 3 load 4 load 1

选择load 1

所以这里我们的指令是

load

。。。

一直到结束，便是这样一边up一边更新latency，一边选择指令。

需要注意的是latency-first的意思是首先判断latency，当latency一样的时候参考其他数据，比如当前寄存器压力，路径深度等等。

后注：

RA之后，寄存器是有限的，还记得bigEUsize吗？EU越大，总寄存器就越多，可并发warp就越多，thread并发度就越高。

thread并发度和Instruction并发度是需要做出取舍的，可以取中庸，也可以取两边极致，看硬件配置是大核小核，register file大小等

举个例子，现在是simd64，1个thread需要用到40个register，那么一个warp就需要40 * 64个register，每个EU，总的寄存器数量是有限的，warp的并发数就会下来。

这时候会有疑问了，啊这？为什么EU的大小不能达到warp并发的极致呢？也就是warp上限 乘以 每个thread寄存器上限 乘以 simdmode，好问题。据我所见，HW的hardware没能做到这一点，我也不知道是技术限制还是设计所为，我更倾向于前者，可惜我已经离职，我会在下家问下硬件团队这个问题。
